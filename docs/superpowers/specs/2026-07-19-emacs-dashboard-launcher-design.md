# Emacs dashboard launcher + toolbar icons — design

Date: 2026-07-19
Status: approved (design), pending implementation plan

## Purpose

A tap-first "home base" for this config that works identically on desktop and
Android touch:

1. A **dashboard** (via `dashboard.el`) shown at startup and re-openable, with
   recents/agenda/bookmarks widgets **plus** a row of clickable buttons that run
   common actions (pull repo, tangle, reload, visit config.org / mindwtr.org,
   capture, agenda, browse notes). Tapping a button runs the action — no
   keyboard, no confirmation except for network/destructive actions.
2. **Meaningful Material icons** on the five Android on-screen toolbar quick
   slots (replacing the numbered ①–⑥ icons), with `dashboard-open` on slot ①.

Hard requirement driving the design: **tap to run, no keyboard input, no
confirmation except network/destructive.** This is why `dashboard.el`'s
clickable navigator buttons (native `mouse-1`/tap, no prompt) are used rather
than an org file of `elisp:` links (which prompt on every follow) or `enlight`
(keyboard-key driven).

## Delivery: two separate commits

- **Commit 1 — Dashboard.** New `config.org` "Dashboard" section (replaces the
  currently `:disabled` Hyperbole/Dashboard block, ~line 2646) → `post-init.el`.
- **Commit 2 — Toolbar slots.** Refactor the Android toolbar slots in the
  `config.org` android-support section → `srijan-lisp/android-support.el`.

Commit 1 first: the toolbar's slot ① binds `dashboard-open`, which `dashboard.el`
(commit 1) provides.

## Context

- `config.org` tangles to `.el` files (gitignored); edits are made in
  `config.org`, then re-tangled. `minimal-emacs-user-directory` is the repo root
  (`~/.emacs.d/`); `user-emacs-directory` is redirected to `var/`.
- Package management: Elpaca + use-package with `use-package-always-defer t` —
  so `dashboard` needs `:demand t` to load at startup.
- `my-phone-p` = Android. `initial-buffer-choice` is currently unset at top
  level. The one `(setq initial-buffer-choice nil)` in `config.org` lives inside
  `my/command-line-diff` (the `-diff` command-line switch handler), so it does
  **not** clobber the dashboard at normal startup — and correctly suppresses it
  for `emacs -diff a b`. Verified; no change needed there.
- Existing helpers: `my-reload-emacs-configuration`, `C-x ,` opens config.org,
  `mindwtr-file` = `(sj/sync "docs/org/mindwtr.org")`, `sj/sync` expands paths
  under `sj/sync-dir`.
- Android toolbar lives in `srijan-lisp/android-support.el` (tangled). Fixed
  items already bind real commands directly (e.g. `consult-buffer`); the six
  quick slots currently use `numeric-N-circle-outline` icons + wrapper commands
  generated by a macro, indirecting through the
  `android-support-global-tool-bar-custom-commands` defcustom (a flat list).
- `material-pbm-icons` ships the full ~7,400-icon Material Design Icons set as
  PBM bitmaps, named by MDI slug (verified). All icons below exist.

## Commit 1 — Dashboard

### Package

```elisp
(use-package dashboard
  :ensure (:host github :repo "emacs-dashboard/dashboard")
  :demand t
  :custom
  (dashboard-items '((agenda . 5) (recents . 5) (bookmarks . 5)))
  (dashboard-center-content t)
  (dashboard-set-footer nil)
  :config
  ;; ... navigator + icon setup below ...
  (dashboard-setup-startup-hook)
  (setq initial-buffer-choice 'dashboard-open))
```

- `:demand t` is required (global defer would otherwise leave it unloaded at the
  first frame, so it could not be the startup buffer).
- `initial-buffer-choice 'dashboard-open` shows it at startup and works in
  daemon/emacsclient frames (the config runs `(server-start)`).

### Widgets

`dashboard-items = ((agenda . 5) (recents . 5) (bookmarks . 5))`. Agenda draws
from the existing org-agenda files (mindwtr). Recents from recentf.

### Launcher (navigator buttons)

`dashboard-navigator-buttons` — rows of tappable buttons. Format per button is
`(ICON LABEL HELP ACTION FACE PREFIX SUFFIX)`; `ICON` is `nil` (emoji lives in
`LABEL` so no icon-font dependency on Android), `ACTION` is
`(lambda (&rest _) (COMMAND))`.

| Button label | Command | Confirm? |
|---|---|---|
| ⟳ Pull repo | `sj-dashboard-pull` | network → yes |
| ⚙ Tangle | `sj-dashboard-tangle` | no |
| ↻ Reload | `my-reload-emacs-configuration` | no |
| ✎ config.org | `sj-dashboard-visit-config` | no |
| ✎ mindwtr.org | `sj-dashboard-visit-mindwtr` | no |
| ＋ Capture | `org-capture-inbox` | no |
| 🗓 Agenda | `mindwtr-engage` | no |
| 🔍 Notes | `consult-notes` | no |

Layout: two rows (config actions; notes actions) for readable tap targets.

### Action commands (named defuns, reusable from toolbar / M-x)

- `sj-dashboard-pull` — `git pull` in the repo, with confirmation:
  ```elisp
  (defun sj-dashboard-pull ()
    "Pull changes in the Emacs config repo (asks first — network)."
    (interactive)
    (when (sj-dashboard--confirm "Pull Emacs repo changes? ")
      (let ((default-directory minimal-emacs-user-directory))
        (async-shell-command "git pull" "*emacs-repo-pull*"))))
  ```
- `sj-dashboard-tangle` —
  `(org-babel-tangle-file (expand-file-name "config.org" minimal-emacs-user-directory))`
  then a `message` on completion.
- `sj-dashboard-visit-config` —
  `(find-file (expand-file-name "config.org" minimal-emacs-user-directory))`.
- `sj-dashboard-visit-mindwtr` — `(find-file (sj/sync "docs/org/mindwtr.org"))`
  (computed at click time via `sj/sync`, so it tracks `sj/sync-dir`; does not
  depend on `mindwtr-file` being bound yet).
- Reused as-is: `my-reload-emacs-configuration`, `org-capture-inbox`,
  `mindwtr-engage`, `consult-notes`.

### Confirmation (tap-friendly)

```elisp
(defun sj-dashboard--confirm (prompt)
  "Ask PROMPT. Uses a tappable GUI dialog when invoked from a touch/mouse
event on a graphical frame; falls back to `y-or-n-p' in a terminal."
  (let ((use-dialog-box (display-graphic-p)))
    (y-or-n-p prompt)))
```

When a navigator button is tapped, `last-nonmenu-event` is the touch/mouse
event, so `y-or-n-p` pops a clickable Yes/No dialog under `use-dialog-box`
(no keyboard needed). In a terminal it degrades to a normal prompt. Only
`:network`/destructive actions call this; safe actions run on a single tap.

### Icons (startup-safe)

- Desktop: use `nerd-icons` or `all-the-icons` if the feature is available;
  otherwise leave icons off.
- Android: set `dashboard-set-file-icons nil` and `dashboard-set-heading-icons
  nil` to avoid an icon-font dependency; navigator emoji still render.
- All icon setup is guarded so a missing icon package cannot abort startup
  (pattern already used for the phone font / android-support load).

### Touch styling

`dashboard-center-content t`; add buffer line-spacing in the dashboard buffer
so navigator buttons are comfortable tap targets. No landscape-specific layout.

## Commit 2 — Toolbar slots with meaningful icons

Refactor `srijan-lisp/android-support.el` so the quick slots are **data-driven**
and carry their own icons.

### New defcustom (replaces the flat command list)

```elisp
(defcustom android-support-global-tool-bar-slots
  '((dashboard-open                          . "view-dashboard-outline")
    (org-capture-inbox                       . "inbox-arrow-down-outline")
    (mindwtr-engage                          . "calendar-check-outline")
    (denote-journal-new-or-existing-entry    . "notebook-outline")
    (consult-notes                           . "text-box-search-outline"))
  "Quick-command slots on the Android tool bar, as (COMMAND . MDI-ICON-NAME)."
  :type '(alist :key-type function :value-type string)
  :group 'android-support)
```

### Changes

- Remove the old `android-support-global-tool-bar-custom-commands` defcustom, the
  `android-support--define-custom-commands` macro, and its invocation.
- Remove the six `("numeric-N-circle-outline" ...custom-command-N)` lines from
  `android-support-global-tool-bar-items`.
- In `android-support-global-tool-bar-setup`, after adding the fixed items, loop
  `android-support-global-tool-bar-slots` and add each with
  `(tool-bar-add-item ICON COMMAND COMMAND)` — binding the real command directly,
  the same pattern the fixed items already use (e.g. `consult-buffer`). All five
  commands are autoloaded by the time the bar renders (`dashboard-open` from the
  `:demand`ed dashboard; the others from their use-package autoloads).

### Result

Five slots with meaningful icons; slot ⑥ gone; adding/removing/reordering a slot
(icon included) is a one-line edit to the alist.

## Testing

- Re-tangle `config.org`; confirm `post-init.el` and
  `srijan-lisp/android-support.el` regenerate and parse (successful tangle +
  byte-compile with no errors), as for prior changes.
- Batch sanity: `emacs -Q --batch` load of `android-support.el` builds the
  toolbar map from the new alist without error.
- Desktop manual: dashboard appears at startup with agenda/recents; each
  navigator button runs its action on click; Pull shows a Yes/No dialog first.
- Android manual (user): dashboard is the startup buffer; navigator buttons run
  on tap with no keyboard; Pull shows a tappable dialog; toolbar slots ①–⑤ show
  the new icons and launch their commands.

## Out of scope

- Projects/registers widgets (can be added to `dashboard-items` later).
- Reworking the fixed toolbar items (arrows, prefixes, chords) — unchanged.
- Any new dependency beyond `dashboard.el` (already-present icon packages are
  used only if available).
